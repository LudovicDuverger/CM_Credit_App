# Refactored Architecture

## Overview

This repository runs as a two-part web application:

- `maestro-ui/`: active React + Vite frontend for credit analysts
- `backend/`: active Node.js + Express backend in TypeScript

The backend migration to the UiPath TypeScript SDK is complete with an SDK-first approach and a thin raw HTTP fallback for tenant-specific gaps.

## Required Reading (Start Here)

For any developer or coding agent intervening on this project, follow this order before making changes:

1. Read this file (`architecture.md`) end-to-end.
2. Read `uipath-typescript-SDK-context.txt` for the SDK API surface and usage patterns.
3. Only then inspect implementation files in `backend/` and `maestro-ui/`.

Contribution rule:

- If a change touches UiPath integration (`backend/lib/uipath-sdk.ts`, `backend/lib/entity-operations.ts`, `backend/routes/*`), `uipath-typescript-SDK-context.txt` is required context.

## Migration Status

Backend migration objectives now implemented:

- backend source moved from `.js` to `.ts`
- API contracts kept stable for frontend consumers
- UiPath integration moved to `@uipath/uipath-typescript` where supported
- both auth modes preserved:
  - incoming bearer token
  - backend client-credentials fallback
- raw HTTP retained only as fallback behavior

## Active Runtime

### Frontend

Entrypoints:

- `maestro-ui/src/main.tsx`
- `maestro-ui/src/App.tsx`

Key points:

- React + React Router + Vite
- fetch-based service layer in `src/services/*`
- frontend talks to backend API (`VITE_API_BASE_URL`, default `http://localhost:3001/api`)

### Backend

Runtime composition:

- `backend/server.ts`: process startup and mode logging
- `backend/app.ts`: Express app creation, middleware, route mounting, optional static frontend serving
- `backend/routes/*`: API route contracts
- `backend/lib/*`: UiPath SDK context, auth + fallback HTTP client, mapping, aggregation, Data Fabric operations

## Backend Module Responsibilities

### `backend/server.ts`

- boots server on `PORT` (default `3001`)
- logs runtime mode (UiPath or mock) and auth mode

### `backend/app.ts`

- configures middleware (`cors`, JSON parser)
- mounts `/api` routes
- serves `maestro-ui/dist` when built

### `backend/config/uipath.ts`

Centralized UiPath configuration:

- base URL, org, tenant
- folder key and optional `UIPATH_FOLDER_ID`
- client credentials and scopes
- target process/case model and entity names

Exposes mode helpers:

- `hasUiPathBaseConfig()`
- `hasClientCredentials()`

### `backend/lib/uipath-sdk.ts`

SDK context factory:

- creates per-request SDK instance using `secret` token auth
- exposes typed clients for `Tasks`, `Entities`, `Cases`, `CaseInstances`
- resolves task `folderId` from task metadata with `UIPATH_FOLDER_ID` fallback

### `backend/lib/uipath-client.ts`

Low-level helpers retained for fallback paths:

- bearer extraction from request headers
- client-credentials token retrieval + caching
- URL/header builders
- raw request wrappers for fallback behavior

### `backend/lib/fetch-uipath-data.ts`

Case aggregation layer:

- SDK-first reads from `Cases`, `CaseInstances`, `Entities`
- resolves main/document entities
- enriches list/detail responses with stage/task/activity/document data
- preserves list/detail API response shape

### `backend/lib/case-processors.ts`

Case transformation utilities:

- stages normalization
- action tasks normalization
- activity timeline generation
- case detail assembly

SDK-first calls:

- `CaseInstances.getStages`
- `CaseInstances.getExecutionHistory`
- `CaseInstances.getActionTasks`

raw fallback remains for compatibility.

### `backend/lib/entity-operations.ts`

Data Fabric operations:

- read records
- create/update/delete records
- upload attachments
- document field mapping and case-id update flow

SDK-first methods:

- `Entities.getAllRecords`
- `Entities.insertRecordById`
- `Entities.updateRecordById`
- `Entities.deleteRecordsById`
- `Entities.uploadAttachment`
- `Entities.downloadAttachment` (used in route layer)

### `backend/routes/system.ts`

- `GET /api/health`
- `GET /api/source`

### `backend/routes/cases.ts`

- `POST /api/debug/uipath-data`
- `GET /api/cases`
- `GET /api/cases/:id`

Behavior:

- mock mode uses `backend/mock/data.ts`
- UiPath mode uses SDK-first aggregation

### `backend/routes/tasks.ts`

- `GET /api/tasks/:taskId`
- `GET /api/tasks/:taskId/form`
- `POST /api/tasks/:taskId/assign-self`
- `POST /api/tasks/:taskId/complete`

Behavior:

- task read/assign/complete use SDK first
- `/form` falls back to raw forms endpoint if SDK payload has no form data
- completion resolves `folderId` from task metadata, then `UIPATH_FOLDER_ID`

### `backend/routes/documents.ts`

- `GET /api/documents/:recordId`
- `POST /api/cases/:caseId/documents`
- `DELETE /api/documents/:recordId`

Behavior:

- SDK-first attachment operations with fallback
- stable frontend contract for document URLs and metadata

### `backend/routes/loan-requests.ts`

- `POST /api/loan-requests`

Behavior:

- parses multipart payload
- creates main case and document records
- uploads attachments
- returns created case + upload results

### `backend/routes/oauth.ts`

- `POST /api/oauth/token`

Used for PKCE code exchange during frontend callback flow.

## Integration Strategy: SDK-first with Fallback

The backend intentionally uses a hybrid strategy:

- default path: UiPath TypeScript SDK for supported features
- fallback path: raw HTTP only when SDK behavior is missing or tenant-specific edge behavior requires it

This keeps code strongly typed while preserving robustness across environments.

## Authentication Strategy

Both auth modes are supported:

- frontend bearer token (`Authorization: Bearer <token>`)
- backend client credentials (when no bearer token is provided and credentials are configured)

Concurrency safety:

- SDK context is created per request with request-scoped token
- parallel requests with different bearer tokens do not share mutable auth state

## Folder Resolution Strategy for Task Completion

Task completion requires a numeric `folderId`.

Resolution order:

1. `folderId` from task metadata
2. `FolderId` from task metadata
3. organization unit metadata when available
4. env fallback `UIPATH_FOLDER_ID`

If no `folderId` is resolved, backend returns a clear `502` with guidance.

## API Compatibility

The migration keeps public route contracts stable for frontend clients:

- route paths unchanged
- response shapes preserved
- frontend service modules continue to call the same API endpoints

## Main Application Flows

### 1. Authentication Flow

1. User opens `/login`
2. Frontend starts UiPath OAuth PKCE flow
3. UiPath redirects to `/oauth-callback`
4. Frontend sends authorization code to `POST /api/oauth/token`
5. Backend exchanges code against UiPath Identity
6. Frontend stores access token and unlocks protected routes

### 2. Case List Flow

1. Frontend calls `GET /api/cases`
2. Backend resolves auth token
3. Backend fetches and maps UiPath case data (or mock data)
4. Frontend renders submissions/dashboard list views

### 3. Case Detail Flow

1. Frontend calls `GET /api/cases/:id`
2. Backend enriches case with stages, tasks, chronology, documents
3. Frontend renders detail workspace

### 4. Document Flow

1. Frontend opens or uploads a case document
2. Backend resolves Data Fabric entity and attachment
3. Backend streams/downloads attachment or writes upload
4. Frontend updates preview/table state

### 5. In-App Task Validation Flow

1. Frontend navigates to `/cases/:id/tasks/:taskId`
2. Frontend calls assign-self endpoint
3. Frontend fetches task metadata + form
4. Backend serves SDK task payload or fallback form payload
5. Frontend submits completion with selected action and form data
6. Backend completes task via SDK with resolved `folderId`

#### AgentAnalysis field flow

`AgentAnalysis` is produced upstream by UiPath automation and is not generated by this backend.
The backend transports it as part of the task form data and completion payload.

### 6. Loan Request Creation Flow

1. Frontend submits multipart payload to `POST /api/loan-requests`
2. Backend validates payload
3. Backend writes main case entity record
4. Backend writes document records + uploads attachments
5. Backend returns case id and upload status

## Runtime Modes

### Mock Mode

Enabled when UiPath base config is missing or `USE_MOCK_DATA=true`.

Used for:

- local demos
- UI validation without live UiPath dependency

### UiPath Mode

Enabled when base UiPath config is present.

Auth is bearer-first with client-credentials fallback.

## Static Frontend Serving

Backend serves `maestro-ui/dist` when available.

In development:

- frontend and backend usually run separately

In production-style mode:

- build frontend (`maestro-ui/dist`)
- backend serves the built assets
